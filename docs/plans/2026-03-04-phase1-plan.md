# Phase 1: Architecture Migration + Infrastructure

**Date**: 2026-03-04
**Status**: Approved
**Parent**: [Skills Optimization Design](./2026-03-03-skills-optimization-design.md)
**Issue**: [MARS-3517](https://linear.app/marswave/issue/MARS-3517)

## Scope

Phase 1 delivers the docs-only architecture migration and skill decomposition. Four workstreams:

1. **shared/ 基础层** — API reference, authentication, common patterns
2. **Base skills** — podcast, explainer, speech, image-gen (decomposed from ListenHub)
3. **content-parser** — URL content extraction skill
4. **Migration cleanup** — deprecate listenhub/, update README

**Deferred to Phase 2:** style system.

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| API call method | curl via Bash | Simplest, no extra tools needed. AI reads docs and constructs curl commands |
| API doc source | Official ListenHub API docs | User provides; more accurate than reverse-engineering from scripts |
| API doc structure | Centralized `skills/shared/api-reference.md` | Single source of truth, no duplication across skills |
| Skill references/ | Keep for domain knowledge | Per skill-creator pattern: on-demand context loading |
| Repo structure | `skills/<name>/` | Preserves existing convention |
| Interaction model | AskUserQuestion options | All enumerable params use multiple choice; free text only for topic/content/URL |

## Execution Order

Bottom-up: complete all infrastructure first, then build all skills.

```
Step 1: skills/shared/ — API reference, authentication, common patterns
Step 2: skills/podcast/, explainer/, speech/, image-gen/ — all 4 base skills
Step 3: skills/content-parser/ — content extraction skill
Step 4: Migration cleanup — deprecate listenhub/, update README, CHANGELOG
```

## Deliverable 1: shared/ 基础层

### api-reference.md

Complete ListenHub API reference, written from official API documentation. Organized by functional area:

1. **Speakers** — `GET /speakers/list`
2. **Podcast** — `POST /podcast/episodes`, `POST .../text-content`, `POST .../audio`, `GET /podcast/episodes/{id}`
3. **Explainer/Storybook** — `POST /storybook/episodes`, `POST .../video`, `GET /storybook/episodes/{id}`
4. **FlowSpeech** — `POST /flow-speech/episodes`, `GET /flow-speech/episodes/{id}`
5. **Speech** — `POST /speech`
6. **Image Generation** — `POST /images/generation` (Labnana API)
7. **Content Parsing** — endpoint TBD (pending official docs)

Each endpoint includes: HTTP method + path, request parameters (required/optional, type, description), curl example, response structure (JSON schema + example), error codes.

### authentication.md

- Environment variable `LISTENHUB_API_KEY` in `~/.zshrc` or `~/.bashrc`
- Base URL: `https://api.marswave.ai/openapi/v1`
- Image generation URL: `https://api.labnana.com/openapi/v1`
- Headers: `Authorization: Bearer $LISTENHUB_API_KEY`, `Client-Id: PJBkELS1o_q9nJ~NzF2_Fmr21TNX&~eoJR49FFdFhD3U`
- How to obtain an API key (Pro plan required)

### common-patterns.md

Reusable patterns extracted from existing scripts:

**Async polling:**
- Submit task → get ID → poll status (interval 10s, max 30 polls / 300s)
- Status field parsing with fallback
- Exit conditions: completed, failed, timeout

**Error handling:**
- HTTP status codes and meanings
- 429 rate limit: exponential backoff
- Network errors: retry up to 3 times
- Standard response structure: `{code, data, message}`

**Input validation:**
- URL format requirements
- Content length limits (10K chars for TTS text)
- Supported languages: zh, en
- ID format: alphanumeric + hyphen + underscore

**Interactive parameter collection:**
- Use AskUserQuestion for all enumerable parameters
- Present options with descriptions (not just labels)
- Free text only for: topic, content, URL, image prompt
- One question at a time when there are dependencies (e.g., speaker list depends on language)
- Batch independent questions when possible (e.g., resolution + aspect ratio)

### podcast/

**Trigger:** "做播客"、"podcast"、"播客"、"录一期节目"

**Interaction Flow:**
1. 主题/内容来源 — free text or URL
2. 模式 → [快速 Quick / 深度 Deep / 辩论 Debate]
3. 语言 → [中文 / English]
4. 主播数量 → [1人独白 / 2人对话]
5. 主播选择 → call speakers API, present list as options
6. 参考资料 → [有 URL / 有文本 / 无]
7. 生成方式 → [一步生成 / 两步审阅（先文本再音频）]

**Workflow:**
- One-step: POST /podcast/episodes → poll → present result
- Two-step: POST .../text-content → present scripts for review → user edits → POST .../audio → poll → present result

**references/mode-guide.md:** Explains quick/deep/debate mode differences, when to recommend each, content depth expectations.

**references/speaker-selection.md:** Speaker characteristics, pairing recommendations, voice style descriptions.

**Composability:** Invoked by content-planner (Phase 3).

### explainer/

**Trigger:** "解说视频"、"explainer video"、"解释一下XX（视频形式）"

**Interaction Flow:**
1. 主题/内容 — free text
2. 语言 → [中文 / English]
3. 风格 → [信息型 Info / 叙事型 Story]
4. 主播选择 → single speaker from API list
5. 输出 → [仅文本脚本 / 文本+视频]

**Workflow:**
- POST /storybook/episodes → poll → present text/script
- If video requested: POST .../video → poll → present video URL

**references/video-guide.md:** Video style options, pacing guidance, visual storytelling tips.

**Composability:** May invoke speech (for voiceover). Invoked by content-planner (Phase 3).

### speech/

**Trigger:** "朗读这段"、"配音"、"TTS"、"语音合成"、"text to speech"

**Interaction Flow:**
1. 输入类型 → [文本 / URL]
2. 内容 — free text or URL
3. 语言 → [中文 / English]
4. 模式 → [直接朗读 Direct / 智能处理 Smart（修正语法标点）]
5. 主播选择 → single speaker
6. 如果多人 → 引导编写 scripts JSON（speaker assignment per segment）

**Two API paths:**
- Single speaker (FlowSpeech): POST /flow-speech/episodes → poll
- Multi-speaker scripts: POST /speech → poll

**references/tts-guide.md:** FlowSpeech vs Speech API differences, when to use each, multi-speaker script format and authoring tips.

**Composability:** Invoked by explainer, douyin-script (Phase 2), youtube (Phase 2).

### image-gen/

**Trigger:** "生成图片"、"画一张"、"AI图"、"generate image"、"配图"

**Interaction Flow:**
1. 图片描述 — free text (AI assists with prompt optimization)
2. 分辨率 → [1K / 2K / 4K]
3. 比例 → [16:9 / 1:1 / 9:16 / 2:3 / 3:2 / 3:4 / 4:3 / 21:9]
4. 参考图 → [有 URL(s) / 无]

**Workflow:**
- POST /images/generation → receive base64 image → decode → save to local file
- File saved to `$LISTENHUB_OUTPUT_DIR` (default: ~/Downloads)
- Unique filename: `listenhub-YYYYMMDD-HHMMSS-XXXX.jpg`

**Special handling:**
- Base64 decoding in bash (base64 -d on Linux, base64 -D on macOS)
- Retry logic for rate limits (429)

**references/prompt-guide.md:** Image prompt writing techniques, style keywords, composition tips, reference image usage.

**Composability:** Invoked by wechat-article, xiaohongshu, twitter, instagram, youtube (Phase 2).

## Deliverable 3: content-parser/

**Trigger:** User provides a URL to extract content from; another skill needs to parse source material.

**When NOT to Use:** User already has text content; user wants to generate content (use platform skills).

**Interaction Flow:**
1. URL — free text
2. Confirm platform recognition → display detected platform and content type

**Workflow:**
- Call ListenHub content parsing API → receive normalized JSON
- Present structured result: platform, type, title, author, content, media, metadata

**Output format:**
```json
{
  "platform": "twitter",
  "type": "thread",
  "title": null,
  "author": "@username",
  "content": "...",
  "media": [],
  "metadata": {
    "published_at": "...",
    "engagement": {},
    "tags": []
  }
}
```

**references/supported-platforms.md:** Platform URL patterns, content types per platform, parsing quirks and limitations.

**Composability:** Invokes nothing (direct API calls). Invoked by all platform skills (Phase 2), content-planner (Phase 3).
